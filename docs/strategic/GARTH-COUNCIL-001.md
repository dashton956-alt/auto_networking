# GARTH — AI Council
## Hallucination Reduction & AI Validation Framework

**Document:** GARTH-COUNCIL-001  
**Platform:** GARTH — Governed Autonomous Routing, Telemetry & Healing  
**Version:** 0.1.0  
**Status:** Active  
**Applies to:** All AI-assisted decision points in the GARTH prototype  

---

## Purpose

GARTH is an autonomous networking platform. Incorrect decisions do not produce wrong answers on a screen — they produce misconfigured networks, dropped traffic, and outages. Hallucination is not a UX problem here. It is a safety problem.

This document defines the AI Council — a structured multi-model validation pattern that GARTH uses to reduce hallucination risk at every AI-assisted decision point. The council is not a single AI making a decision. It is a structured process where multiple perspectives challenge, verify, and must agree before any output is trusted.

The AI Council applies during two distinct contexts:

1. **Build time** — when AI coding agents (Claude Code, Copilot, etc.) are generating the prototype codebase
2. **Runtime** — when AI-assisted components inside GARTH are generating policy suggestions, risk assessments, or feedback tuning recommendations

---

## Core Principle

> No AI output that affects a network state, policy evaluation, or risk score is accepted on first generation. Every such output must pass a structured challenge before it is used.

This does not mean slowing everything down. It means having explicit validation gates — lightweight, fast, and automated where possible — that catch errors before they propagate downstream.

## Part 1 — Build Time AI Council

### What It Covers

When using spec-kit with Claude Code (or any AI coding agent) to build the GARTH prototype, the following output types are high-risk for hallucination and must be validated before being accepted into the codebase:

| Output type | Hallucination risk | Why dangerous |
|---|---|---|
| Policy evaluation logic | High | Wrong logic silently passes bad intents |
| Risk scoring formula | High | Wrong scores lead to wrong execution modes |
| Decision tree logic | High | Wrong branches select wrong actions |
| Schema definitions | Medium | Downstream validation fails silently |
| Rollback handler logic | High | Failed rollback leaves network in broken state |
| Audit record writes | Medium | Missing writes create compliance gaps |
| Governance mode gate rules | High | Wrong rules bypass human approval |

### The Build-Time Council Pattern

For every high-risk output generated by an AI coding agent, apply this three-role challenge before merging:

---

#### Role 1 — The Generator
The AI coding agent that produced the output. It presents its work.

**Prompt to use after any high-risk module is generated:**
```
You have just generated [module name]. Before I accept this, I need you to act as a 
critical reviewer, not the author. Read what you just wrote and answer:

1. Is the logic in this function deterministic? Could it ever produce different 
   outputs for the same inputs?
2. Is there any code path that could silently succeed when it should fail?
3. Does every action-triggering path write to the audit log before returning?
4. Is there a rollback handler for every executor, and does it actually reverse 
   the action completely?
5. Are there any assumptions baked in that are not validated by the schema?

List every concern you find, including minor ones. Do not defend the code — find its flaws.
```

---

#### Role 2 — The Adversary
A second AI prompt that attempts to break the output by finding edge cases, logic errors, and missing constraints.

**Prompt to use:**
```
You are a hostile network engineer who does not trust this code. Your job is to 
find scenarios where it produces an incorrect result, skips a safety check, or 
allows an action that should be blocked.

Here is the code: [paste module]

Here is what it is supposed to do: [paste acceptance criteria from spec.md]

Find at least 3 scenarios where this code does the wrong thing. Be specific — 
give the exact input that triggers the failure and describe the incorrect output.
If you cannot find 3 genuine failures, say so and explain why the code is sound.
```

---

#### Role 3 — The Verifier
A structured checklist that must be completed before the output is accepted. This is not AI-generated — it is a human-reviewed gate.

**Checklist (must all be checked before merging any high-risk module):**

**Determinism**
- [ ] Function produces identical output for identical inputs (verified by running the same test case 3 times)
- [ ] No `random`, `uuid4` (outside of ID generation), `datetime.now()` (outside of audit timestamps), or external API calls in evaluation logic
- [ ] All enum values used are declared in schemas — no magic strings

**Safety**
- [ ] Every execution path that triggers an action writes an audit record first
- [ ] Every action executor has a `rollback()` method that is independently callable
- [ ] Rollback has been manually traced to confirm it reverses the action completely
- [ ] No code path returns `success` when the underlying operation has not completed

**Schema compliance**
- [ ] All inputs validated against Pydantic models before processing begins
- [ ] All outputs conform to their declared response schema
- [ ] Invalid inputs return 422 with structured field-level errors, not 500

**Governance**
- [ ] No action execution path bypasses the governance mode gate
- [ ] `isolate_segment` action type always resolves to `manual_review` mode
- [ ] `block` mode never proceeds to execution under any condition

**Explainability**
- [ ] Every decision object contains a `reasoning_chain` with at least one entry per decision step
- [ ] The `/why` endpoint can be called for this decision and returns human-readable output

---

### Build-Time Council: When to Apply

Apply the full three-role challenge when generating or modifying:

- `src/anif/policy/` — any file
- `src/anif/risk/` — any file
- `src/anif/decision/` — any file
- `src/anif/governance/` — any file
- `src/anif/actions/` — any executor or rollback handler
- `src/anif/audit/` — the write path specifically
- Any schema file in `schemas/`

Apply the Verifier checklist only (skip Generator and Adversary) for:
- API router files
- Pydantic model definitions
- Test files
- Configuration files

---

### Build-Time Hallucination Triggers to Watch For

These are the most common hallucination patterns observed when AI agents generate networking and policy code. Flag immediately if seen:

| Pattern | What to look for | What to do |
|---|---|---|
| **Invented policy names** | Policy names in code that don't exist in `schemas/policy_schema.yml` | Reject. Add to schema or remove from code. |
| **Assumed network state** | Code that assumes `healthy` network state without reading it from input | Reject. Network state must always be an explicit input parameter. |
| **Silent success on missing data** | Functions that return a passing result when a required field is absent | Reject. Missing required fields must raise a validation error. |
| **Partial rollback** | Rollback handlers that reverse some but not all changes | Reject. Rollback must be atomic or explicitly declared as partial with an escalation path. |
| **Confidence without basis** | Risk or confidence scores that are returned without a justification list | Reject. Every numeric score must have a `justification` field populated. |
| **Bypassed governance** | Execution code that checks governance result but proceeds on `null` or missing result | Reject. Absent governance result must default to `block`. |
| **Hardcoded thresholds** | Risk thresholds (e.g. `>= 70`) hardcoded in logic rather than read from config | Reject. All thresholds must be configurable. |
| **Invented API responses** | Mock adapter responses that invent fields not in the schema | Reject. Mock responses must be valid instances of the declared response schema. |

## Part 2 — Runtime AI Council

### What It Covers

GARTH's closed-loop feedback module (US-09) and any future AI-assisted policy suggestion components use AI to generate recommendations. These are lower-risk than action execution (they are always human-reviewed before application) but still require validation to prevent garbage suggestions from degrading the system over time.

### The Runtime Council Pattern

Runtime AI outputs in GARTH fall into two categories:

**Category A — Advisory only (suggestions, analysis, tuning recommendations)**
These are never auto-applied. They are presented to a human operator for review. Validation is lightweight.

**Category B — Inputs to deterministic systems (intent parsing, natural language to structured intent)**
These feed into the deterministic policy and risk engines. Validation is strict.

---

#### Category A Validation — Advisory Outputs

All advisory outputs from AI components must include:

```json
{
  "suggestion_id": "uuid",
  "type": "threshold|policy_condition|risk_weight",
  "current_value": {},
  "suggested_value": {},
  "rationale": "human-readable explanation",
  "evidence": ["audit_record_id_1", "audit_record_id_2"],
  "confidence": "low|medium|high",
  "council_checks": {
    "evidence_count": 0,
    "minimum_evidence_required": 10,
    "evidence_threshold_met": false,
    "contradicting_evidence_count": 0,
    "net_confidence_basis": "string"
  }
}
```

**Validation rules for advisory outputs:**
- `confidence: high` requires `evidence_count ≥ 25` audit records supporting the suggestion
- `confidence: medium` requires `evidence_count ≥ 10`
- `confidence: low` requires `evidence_count ≥ 3`
- Any suggestion with `contradicting_evidence_count > evidence_count / 3` is downgraded one confidence level
- Suggestions with zero evidence references are rejected entirely — no evidence, no suggestion
- Rationale must reference specific patterns observed, not generic statements

---

#### Category B Validation — Structured Intent Inputs

If GARTH ever accepts natural language intent (post-prototype feature), the AI parsing layer must apply a three-stage council before passing output to the deterministic pipeline:

**Stage 1 — Parse:** AI converts natural language to structured intent JSON  
**Stage 2 — Reflect:** Same AI (new context window) is given the structured intent and asked to convert it back to natural language and confirm it matches the original request  
**Stage 3 — Validate:** Structured intent is run through the standard Pydantic schema validation — if it fails schema, the whole parse is rejected regardless of Stage 2 result  

If Stage 2 produces a description that materially differs from the original request, the parse is rejected and the user is asked to restate their intent in more structured terms.

This round-trip validation catches the most common Category B hallucination: confident but wrong field mapping.

## Part 3 — Council Governance

### Who Owns the AI Council

The AI Council process is owned by the GARTH Working Group. Any change to the council checklist, challenge prompts, or validation rules must be:

1. Proposed as an issue in the repository
2. Reviewed by at least one working group member
3. Tested against at least 3 real examples before merging
4. Recorded in the CHANGELOG

### Council Outputs Are Logged

Every time the AI Council is applied:
- The Generator, Adversary, and Verifier outputs are saved to `docs/council-logs/{date}-{module}.md`
- The Verifier checklist is committed alongside the module it validated
- Any rejected outputs are saved to `docs/council-logs/rejected/` with the reason for rejection

This creates a learning record of what the AI got wrong, which informs future prompt improvements.

### Escalation

If the Generator and Adversary produce contradictory findings that cannot be resolved by the Verifier checklist, escalate to a human engineer before merging. Do not average the outputs. Do not proceed on ambiguity. This is P-07 (Fail Safe) applied to the build process itself.

### Build-Time Challenge Prompts (copy-paste ready)

**Generator self-review:**
```
Act as a critical reviewer of the code you just wrote. Find every place where:
- The logic could be non-deterministic
- An action could proceed without an audit record
- A failure could be silently swallowed
- A rollback path is missing or incomplete
- An assumption is made that isn't validated by the input schema
List all findings. Do not defend the code.
```

**Adversary challenge:**
```
You are a hostile reviewer. Given this code and its acceptance criteria, find 3 
specific inputs that produce incorrect outputs. Be exact — give the input and 
describe the wrong output. If the code is sound, say so and explain why.
```

**Schema drift check:**
```
Compare this code against the schema files in /schemas/. List every field name,
enum value, or type used in the code that does not exist in the schemas.
Any discrepancy is a bug — list them all.
```
