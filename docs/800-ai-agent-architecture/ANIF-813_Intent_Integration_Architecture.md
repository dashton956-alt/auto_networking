## Abstract

This document defines the normative architecture for ingesting intents from external sources into the ANIF pipeline. External intent sources include OSS/BSS systems, monitoring platforms, and project management tools. Every external intent source MUST connect through a registered integration adapter. Intents from external sources are subject to the same validation, policy evaluation, and approval process as operator-authored intents. A quality feedback loop scores external intent sources by their execution success rate; sources whose intents fail at a rate above 15% MUST be flagged to the Intent Engineer Agent for correction template generation.

---

## 1. Introduction

### 1.1 Purpose

In production deployments, the majority of intents may originate from automated systems — monitoring platforms raising capacity intents, OSS systems submitting provisioning intents, or project management tools triggering planned maintenance intents. This document defines the normative architecture that governs how these external intent sources are connected, validated, and quality-managed.

### 1.2 Scope

This document covers:

- External intent source categories and their integration adapter requirements
- Quality feedback loop: scoring external sources by execution outcome
- High-failure-rate source detection and correction template generation
- The relationship between external intent submission and the intent lifecycle (ANIF-811)

### 1.3 Out of Scope

This document does not cover:

- Intent authoring format and schema (see ANIF-301)
- Intent lifecycle state machine (see ANIF-811)
- Process system adapters for ITSM, CMDB, and project tools (see ANIF-810)
- Agent tool integration for network execution (see ANIF-814)

### 1.4 Intended Audience

- Integration engineers connecting external systems to ANIF
- AI engineers implementing the Intent Engineer Agent
- Governance officers monitoring intent source quality
- Platform architects designing multi-source intent pipelines

## 3. Terms and Definitions

| Term | Definition |
|---|---|
| External intent source | Any system other than a human operator that submits intents to the ANIF pipeline |
| Intent integration adapter | A registered adapter that receives intent submissions from an external source, validates format, and forwards conformant intents to the pipeline |
| Source quality score | A metric (0.0–1.0) representing the proportion of intents from a source that reached COMPLETED state within their TTL |
| Correction template | A structured guide generated by the Intent Engineer Agent describing common authoring errors for a specific external source |
| Intent Engineer Agent | The agent role responsible for intent quality improvement and correction template management |

## 4. External Intent Source Categories

### 4.1 OSS/BSS Systems

Operations Support Systems and Business Support Systems submit intents related to service provisioning, service modification, and capacity management.

Typical intent types from OSS/BSS:
- Service activation intents (new customer circuit, VPN provisioning)
- Service modification intents (bandwidth upgrade, QoS reclassification)
- Service deactivation intents (circuit decommission)

### 4.2 Monitoring Platforms

Network monitoring and observability platforms submit intents triggered by threshold breaches or anomaly detection.

Typical intent types from monitoring platforms:
- Capacity intents (utilisation above 80% on a specific interface)
- Fault intents (link down event requiring rerouting)
- Performance intents (latency threshold breached, requesting path optimisation)

### 4.3 Project Management Tools

Project management tools submit intents related to planned maintenance and scheduled changes.

Typical intent types from project management tools:
- Maintenance window intents (bring down segment for upgrade)
- Scheduled rollout intents (deploy new configuration at planned time)
- Drain intents (gracefully redirect traffic before planned work)

## 5. Integration Adapter Requirements

### 5.1 Mandatory Registration

Every external intent source MUST connect to the ANIF pipeline through a registered intent integration adapter. Direct API submissions to the pipeline from unregistered sources MUST be rejected.

### 5.2 Adapter Registration Fields

| Field | Description |
|---|---|
| `source_id` | Unique identifier for the external source |
| `source_type` | One of: `oss_bss`, `monitoring`, `project_management` |
| `source_system` | Name and version of the external system |
| `permitted_intent_types` | List of intent types this source is authorised to submit |
| `max_intents_per_hour` | Rate limit for intent submissions from this source |
| `schema_version` | ANIF-301 intent schema version the source submits against |

### 5.3 Format Validation

On receipt of an intent submission, the adapter MUST:

1. Validate the intent against the ANIF-301 schema version declared in the adapter registration.
2. Verify that the submitted `intent_type` is in the source's `permitted_intent_types` list.
3. Verify that the source has not exceeded its `max_intents_per_hour` rate limit.
4. Enrich the intent with `source_id` and `source_type` metadata before forwarding.

Intents that fail any of these checks MUST be rejected with a structured error response and the rejection MUST be logged to the audit trail.

### 5.4 Equivalence with Operator-Authored Intents

Once an external intent passes adapter validation and enters the pipeline, it MUST be processed identically to an operator-authored intent. No special bypasses, reduced validation, or relaxed policy evaluation are permitted for external source intents.

## 6. Quality Feedback Loop

### 6.1 Source Quality Score

The pipeline MUST maintain a quality score for every registered external intent source. The quality score is calculated as:

```
quality_score = COMPLETED_count / (COMPLETED_count + FAILED_count)
```

The score MUST be calculated on a rolling 30-day window. Intents in non-terminal states are excluded from the calculation. The score is updated after every intent reaches a terminal state.

### 6.2 Quality Score Thresholds

| Score Range | Status | Action Required |
|---|---|---|
| 0.85–1.00 | Healthy | No action required |
| 0.70–0.84 | Watch | Log for trend monitoring; no intervention required |
| 0.50–0.69 | Degraded | Notify source administrator; Intent Engineer Agent reviews failure patterns |
| < 0.50 | Critical | Source flagged; Intent Engineer Agent generates correction template; governance notification required |

### 6.3 High-Failure-Rate Trigger

When a source's quality score falls below 0.85, the cause MUST be investigated. When the quality score falls below 0.70:

1. The Intent Engineer Agent MUST be notified.
2. The Intent Engineer Agent MUST analyse the FAILED intent records from the source to identify common failure patterns.

When the quality score falls below 0.50:

1. The Intent Engineer Agent MUST generate a correction template (section 6.4) within 5 business days.
2. The correction template MUST be delivered to the source's registered administrator contact.
3. The source's `max_intents_per_hour` limit MUST be reduced by 50% until quality recovers above 0.70.

### 6.4 Correction Template

A correction template is a structured document generated by the Intent Engineer Agent that describes:

- The most common intent authoring errors observed in the source's failed intents
- Example corrected intent snippets for each error type
- The specific ANIF-301 requirements that the intents are failing against
- Recommended changes to the source system's intent generation logic

Correction templates MUST be versioned and retained in the audit trail as a quality management record.

## 7. Audit Logging

Every external intent submission MUST be logged with:

| Field | Type | Description |
|---|---|---|
| `submission_event_id` | UUID v4 | Unique identifier |
| `source_id` | string | The registered external source |
| `intent_id` | UUID v4 | The intent assigned after validation |
| `outcome` | enum | One of: `accepted`, `rejected` |
| `rejection_reason` | string | Populated if outcome is `rejected` |
| `timestamp` | ISO 8601 | Submission time |

## 8. Conformance Requirements

| ID | Requirement |
|---|---|
| CR-813-01 | All external intent sources MUST connect through a registered intent integration adapter. |
| CR-813-02 | Intents from unregistered sources MUST be rejected. |
| CR-813-03 | External intents MUST be processed identically to operator-authored intents once in the pipeline. |
| CR-813-04 | A quality score MUST be maintained for every registered external intent source on a rolling 30-day basis. |
| CR-813-05 | Sources with quality scores below 0.70 MUST trigger Intent Engineer Agent review. |
| CR-813-06 | Sources with quality scores below 0.50 MUST have their rate limit reduced by 50%. |
| CR-813-07 | The Intent Engineer Agent MUST generate a correction template within 5 business days of a source reaching a Critical quality score. |
| CR-813-08 | Every external intent submission MUST be logged with the fields defined in section 7. |

## 9. Security Considerations

External intent sources are a privileged entry point into the ANIF pipeline. A compromised external system can submit malicious intents at scale. Rate limiting (section 5.2, `max_intents_per_hour`) is the primary throughput control. All external adapter connections MUST use mutual TLS with client certificate authentication — unauthenticated or server-only TLS connections MUST be rejected.

---

## 10. Operational Considerations

Quality score degradation is often gradual. Operators SHOULD monitor score trends rather than waiting for threshold breaches. A score that was 0.95 six months ago and is now 0.80 represents a meaningful degradation trend even though neither value triggers an alert. Monthly quality score review SHOULD be part of normal operations governance.
